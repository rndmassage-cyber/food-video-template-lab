# food-video-template-lab

R&D-репозиторий для проектирования, тестирования и подготовки стабильных шаблонов короткой food-video анимации по исходному изображению блюда.

## Цель проекта

Собрать **template lab** — управляемую лабораторию шаблонов, в которой можно:

- проектировать стабильные motion-шаблоны;
- тестировать их на внутренних изображениях;
- сравнивать результаты;
- отбирать шаблоны для дальнейшего production-использования на сайте.

Главный приоритет проекта: **стабильность, предсказуемость и повторяемость результата**.

## Текущий фокус

На данном этапе проект находится в стадии bootstrap и архитектурной подготовки.

Сейчас задача репозитория:

- зафиксировать рамки V1;
- подготовить структуру проекта;
- подготовить инструкции для Claude Code;
- определить template schema, prompt pack format, testing flow и scoring approach;
- построить базовую R&D-среду без лишней сложности.

## Жёсткие ограничения V1

Пока считать зафиксированными следующие условия:

- используются только **внутренние изображения** из собственного image-pipeline;
- внешние пользовательские загрузки пока не рассматриваются;
- на изображении всегда **одно блюдо**;
- фон всегда **однотонный**;
- блюдо размещается в **safe area** с контролируемым margin;
- допустимы только 3 семейства ракурсов:
  - `top_down`
  - `three_quarter`
  - `hero_side`
- генерация видео выполняется через **Kling 3.0 via Kie AI API**;
- в V1 используется только:
  - `single-shot`
  - `start frame only`
  - `mode=pro`
  - `sound=false`
  - `multi_shots=false`
- пользователь должен иметь возможность выбрать:
  - `template`
  - `aspect_ratio`: `1:1`, `9:16`, `16:9`
  - `duration`
- aspect ratio в V1 обеспечивается через **precompose start frame**, а не через внутреннюю адаптацию Kling.

## Стартовые template families

На первом этапе приоритет только у простых и максимально стабильных шаблонов:

- `orbit_micro`
- `dolly_in_clean`
- `float_parallax`
- `light_accent`

## Что пока сознательно не входит в стартовую реализацию

На старте не приоритизируются:

- multi-shot;
- audio;
- end frame;
- hand / waiter scenes;
- background replacement;
- angle conversion;
- тяжёлый CV / vision analysis, если без него можно обойтись.

## Принцип принятия решений

Если есть выбор между:

- сложным, красивым, но хрупким решением;
- простым, но стабильным решением;

приоритет всегда у **простого и стабильного** решения.

## Базовая структура

- `CLAUDE.md` — проектные инструкции для Claude Code
- `.claude/` — настройки Claude Code для проекта
- `docs/` — документация, roadmap, ADR, runbooks
- `templates/` — спецификации шаблонов, prompt packs, test suites
- `fixtures/` — входные и референсные материалы для тестов
- `runs/` — результаты запусков и R&D-артефакты
- `scripts/` — служебные скрипты проекта

## Правило работы с Claude Code

Claude Code не должен сразу массово создавать файлы и усложнять архитектуру.

Правильный режим работы:

1. сначала прочитать проектный контекст;
2. затем предложить структуру и bootstrap;
3. при любой неясности — задавать вопросы;
4. не принимать спорные архитектурные решения молча;
5. двигаться поэтапно, маленькими проверяемыми шагами.
