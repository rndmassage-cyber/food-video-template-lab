# Project Map — food-video-template-lab

## Что это за проект

**food-video-template-lab** — это R&D-репозиторий для проектирования, тестирования и подготовки стабильных шаблонов короткой food-video анимации по исходному изображению блюда.

Главная цель проекта — собрать **template lab**, в которой можно:

- проектировать template-driven video workflows;
- тестировать стабильность шаблонов;
- сравнивать результаты между шаблонами;
- отбирать надёжные шаблоны для дальнейшего production-использования.

## Главный критерий успеха

Главный критерий успеха проекта:

**стабильность, предсказуемость и повторяемость результата**

Проект не оптимизируется под максимальную креативность любой ценой.  
Приоритет — воспроизводимые и контролируемые шаблоны.

## Что уже зафиксировано как V1

На текущем этапе считать зафиксированными следующие ограничения:

- используются только **внутренние изображения** из собственного image-pipeline;
- внешние пользовательские загрузки пока не рассматриваются;
- на изображении всегда **одно блюдо**;
- фон всегда **однотонный**;
- блюдо находится в **safe area** с контролируемым margin;
- допустимы только 3 семейства ракурсов:
  - `top_down`
  - `three_quarter`
  - `hero_side`
- video generation выполняется через **Kling 3.0 via Kie AI API**;
- в V1 используется только:
  - `single-shot`
  - `start frame only`
  - `mode=pro`
  - `sound=false`
  - `multi_shots=false`
- пользователь должен иметь возможность выбрать:
  - `template`
  - `aspect_ratio`: `1:1`, `9:16`, `16:9`
  - `duration`
- aspect ratio обеспечивается через **precompose start frame**, а не за счёт внутренней адаптации Kling.

## Стартовые template families

На первом этапе фокус только на простых и максимально стабильных шаблонах:

- `orbit_micro`
- `dolly_in_clean`
- `float_parallax`
- `light_accent`

## Что сознательно отложено

Пока не входит в стартовый контур:

- multi-shot;
- audio;
- end frame;
- hand / waiter scenes;
- background replacement;
- angle conversion;
- тяжёлый CV / vision analysis, если без него можно обойтись;
- сложная многоуровневая автоматизация до появления устойчивого V1-контура.

## Логика структуры репозитория

Репозиторий устроен как единый центр R&D-работы.

### Корень проекта

- `README.md` — общее описание проекта и его текущего состояния.
- `CLAUDE.md` — проектные инструкции для Claude Code.
- `.gitignore` — правила игнорирования служебных и generated-файлов.

### `.claude/`

Папка проектных настроек Claude Code.

- `.claude/settings.json` — базовые permissions, hooks и project env для Claude Code.
- `.claude/settings.local.json` — локальные пользовательские overrides, не должны коммититься.

### `docs/`

Папка документации проекта.

- `docs/project-map.md` — этот файл, карта проекта.
- `docs/roadmap.md` — дорожная карта развития проекта.
- `docs/adr/` — архитектурные решения (Architecture Decision Records).
- `docs/runbooks/` — пошаговые рабочие инструкции и операционные сценарии.

### `templates/`

Центральная папка template-driven слоя.

- `templates/specs/` — спецификации шаблонов.
- `templates/prompt-packs/` — prompt packs для шаблонов и сценариев генерации.
- `templates/test-suites/` — наборы тест-кейсов для проверки шаблонов.

В дальнейшем именно `templates/` станет ядром template lab.

### `fixtures/`

Тестовые и референсные материалы.

- `fixtures/raw-inputs/` — исходные входные материалы.
- `fixtures/prepped-images/` — подготовленные precompose/start-frame материалы.
- `fixtures/reference-outputs/` — эталонные и сравнительные результаты.

### `runs/`

Папка для результатов запусков, экспериментов и generated R&D-артефактов.

Содержимое этой папки в основном считается производным и не должно шуметь в git, если это не согласовано отдельно.

### `scripts/`

Служебные скрипты проекта.

На этапе bootstrap эта папка может быть почти пустой.  
Позже здесь могут появиться:

- precompose helpers;
- test runners;
- scoring helpers;
- export / import утилиты;
- локальные automation-скрипты.

## Как принимать решения в этом проекте

Если есть выбор между:

- сложным, красивым, но хрупким решением;
- простым, но стабильным решением;

приоритет всегда у **простого и стабильного** решения.

Проект не должен усложняться раньше времени.

## Как Claude Code должен работать в этом проекте

Claude Code должен:

1. сначала читать проектный контекст;
2. затем формулировать, как он понял задачу;
3. напоминать, что уже зафиксировано;
4. выделять риски и архитектурные развилки;
5. предлагать следующий шаг;
6. при любой неясности задавать вопрос пользователю.

Claude Code не должен:

- молча принимать спорные архитектурные решения;
- массово генерировать файлы без подтверждения;
- вводить лишнюю сложность на раннем этапе;
- путать официальные возможности инструментов с community-наблюдениями.
