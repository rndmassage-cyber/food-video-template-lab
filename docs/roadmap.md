# Roadmap — food-video-template-lab

## Назначение roadmap

Этот документ фиксирует поэтапный план развития проекта **food-video-template-lab**.

Главная цель roadmap — удерживать проект в рамках последовательного и контролируемого внедрения, без преждевременного усложнения.

Ключевой принцип проекта:

**сначала стабильность и предсказуемость, потом расширение возможностей**

## Фаза 2 — Template schema и prompt-pack foundation

### Цель
Создать минимальный и стабильный формат описания шаблонов.

### Задачи
- определить структуру template schema;
- определить обязательные поля template spec;
- определить формат prompt packs;
- определить, как связаны template, prompt pack, test suite и run;
- определить минимальный набор параметров для старта.

### В template schema должны появиться
- идентификатор шаблона;
- family;
- допустимые angles;
- допустимые aspect ratios;
- допустимые durations;
- motion intent;
- prompt strategy;
- precompose assumptions;
- known limitations;
- test requirements;
- scoring criteria;
- status.

### Prompt-pack layer должен зафиксировать
- system-style instruction template;
- motion prompt blocks;
- safe prompt patterns;
- negative / anti-drift guidance;
- template-specific prompt overrides;
- notes по стабильности.

### Критерий завершения
Фаза считается завершённой, когда:
- есть первая рабочая версия template schema;
- есть структура prompt-pack формата;
- можно описать хотя бы 1–2 шаблона в формальном виде;
- эти описания можно проверять и обсуждать как артефакты.

## Фаза 3 — Precompose pipeline design

### Цель
Зафиксировать логику подготовки стартового кадра до генерации видео.

### Почему это важно
В V1 aspect ratio должен обеспечиваться через **precompose start frame**, а не через внутреннюю адаптацию Kling.

Это один из ключевых стабилизирующих механизмов проекта.

### Задачи
- определить входной формат исходного изображения;
- определить safe area и margin rules;
- определить правила приведения к `1:1`, `9:16`, `16:9`;
- определить, как сохраняется блюдо в контролируемой области кадра;
- определить, какие precompose-операции допустимы, а какие нет;
- определить выходной формат подготовленного start frame.

### Важно зафиксировать
- что precompose не должен ломать пропорции блюда;
- что precompose не должен вносить случайную композиционную импровизацию;
- что процесс должен быть повторяемым;
- что правила должны быть пригодны для дальнейшей автоматизации.

### Критерий завершения
Фаза считается завершённой, когда:
- есть ясная спецификация precompose pipeline;
- есть понятные правила для разных aspect ratios;
- можно подготовить эталонные test fixtures.

## Фаза 4 — Kie / Kling adapter planning

### Цель
Подготовить минимальный, стабильный и проверяемый слой интеграции с Kie AI / Kling 3.0.

### Задачи
- описать adapter contract;
- определить обязательные входные параметры;
- определить mapping между template spec и API request;
- определить policy по retries, logging и result capture;
- определить, какие поля response нужно хранить;
- определить ручной и автоматизированный режим запуска.

### Важно
На этом этапе не нужно сразу строить сложную production-интеграцию.

Нужно сначала получить:
- понятный request model;
- минимальный adapter plan;
- правила фиксации результатов запусков;
- воспроизводимый run format.

### Критерий завершения
Фаза считается завершённой, когда:
- есть минимальная спецификация adapter layer;
- известно, какие данные подаются в Kling;
- известно, какие данные нужно сохранять после запуска;
- проект готов к controlled test runs.

## Фаза 5 — Testing и scoring foundation

### Цель
Подготовить систему сравнения и отбора шаблонов.

### Почему это критично
Проект не просто генерирует видео.  
Проект должен уметь **понятно оценивать**, какие шаблоны действительно стабильны.

### Задачи
- определить test suite format;
- определить scoring rubric;
- определить критерии стабильности;
- определить критерии брака;
- определить, как сравниваются результаты между шаблонами;
- определить, как фиксируется verdict по шаблону.

### Возможные scoring-направления
- сохранность формы блюда;
- отсутствие неожиданных деформаций;
- стабильность движения камеры;
- соответствие motion intent;
- отсутствие лишних объектов;
- чистота фона;
- повторяемость результата;
- пригодность к дальнейшему production-use.

### Критерий завершения
Фаза считается завершённой, когда:
- есть первая scoring rubric;
- есть структура test suites;
- можно сравнивать хотя бы несколько run-результатов по единой логике.

## Что не делать раньше времени

До появления стабильного V1-контура не следует преждевременно уходить в:

- multi-shot complexity;
- audio workflows;
- heavy CV-analysis;
- angle conversion;
- background replacement;
- complex scene logic;
- plugin-heavy automation;
- преждевременное разбиение проекта на множество репозиториев;
- сложную orchestration architecture без подтверждённой необходимости.

---

## Логика движения по roadmap

Проект должен двигаться так:

1. bootstrap;
2. фиксация архитектурного каркаса;
3. template schema;
4. precompose logic;
5. adapter planning;
6. testing and scoring;
7. первые стабильные шаблоны;
8. registry и handoff.

Переходить к следующей фазе желательно только тогда, когда предыдущая фаза достаточно зафиксирована и понятна.
